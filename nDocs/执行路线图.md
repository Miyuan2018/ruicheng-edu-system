## 一、文档说明

### 1.1 文档分层与维护策略

本项目文档分为四层，维护策略不同：

| 层级 | 文档 | 维护策略 |
|------|------|----------|
| 执行层 | `nDocs/执行路线图.md` | 状态追踪与任务分解，每次迭代后更新 |
| 需求层 | `nDocs/requirements-v3.0.md` | **活文档**，需求或设计变更时同步更新设计细节章节 |
| 设计层 | `nDocs/backend-api-plan.md`<br>`nDocs/database-design.md`<br>`nDocs/frontend-component-plan.md`<br>`nDocs/error-notebook-design.md`<br>`nDocs/ocr-integration-plan.md`<br>`nDocs/grading-implementation-plan.md`<br>`nDocs/self-study-scheduling-plan.md` | **活文档**，代码变更后必须同步更新 |
| 基线层 | `Raw_Customer_Requirements.md` | **只读基线**，不修改；新功能设计时回溯对照，确保不偏离客户核心诉求 |
| 历史存档 | `docs/requirements-v2.0.md` ~ `v2.4.md`<br>`docs/PROJECT_STATUS.md`<br>`docs/project-summary.md` | 历史参考，不再主动维护 |

**各层关系**：
- 客户基线 → 定义"做什么"（产品边界）
- requirements-v3.0 → 定义"怎么做"（设计细节：API 结构、数据模型、安全方案、部署架构）
- 设计文档 → 定义"技术怎么实现"（端点清单、表结构、组件清单）
- 执行路线图 → 定义"当前做到哪了"（状态追踪、优先级、待办清单）

**维护规则**：
- 新增/修改 API 端点 → 同步更新 `backend-api-plan.md`
- 新增/修改表或字段 → 同步更新 `database-design.md`
- 新增/修改前端页面或组件 → 同步更新 `frontend-component-plan.md`
- 设计决策变更（如 OCR 引擎替换、评分策略调整）→ 同步更新 `requirements-v3.0.md` 对应章节
- 功能实现状态变化 → 在对应设计文档中更新"实现状态"列，在执行路线图中更新任务状态

### 1.2 状态标记

| 标记 | 含义 |
|------|------|
| 已完成 | 代码已合并，构建通过，功能可验证 |
| 进行中 | 已有部分实现，需继续完善 |
| 待实现 | 仅设计/接口存在，核心逻辑未写 |
| 延期 | 明确推迟到后续版本 |

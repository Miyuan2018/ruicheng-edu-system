## 2. V2.0 迭代目标

### 2.1 核心原则
- **先通后优**: 打通所有核心数据流链路, 再优化性能和体验
- **关键路径优先**: 判卷链路 > OCR 链路 > 错题本链路 > 通知链路
- **技术债偿还**: PostgreSQL 迁移、测试覆盖、安全加固

## 5. 架构调整建议

### 5.1 V2.0 推荐架构(模块化单体 → 渐进式微服务)

```
V1.0: 单 FastAPI 应用(42源文件, 模块化)
V2.0: 保持单体 + 提取独立服务
  - judge-worker: 独立判卷工作进程(GPU 独占)
  - ocr-worker: 独立 OCR 工作进程(GPU 独占)
  - notification-worker: 通知发送进程
```

原因: 当前团队规模下, 完全微服务化的运维成本高于收益。将 GPU 密集型任务提取为独立 worker 进程即可满足需求。

### 5.2 数据库策略

```
开发/测试: SQLite(zero-config)
预发布: PostgreSQL(与生产一致)
生产: PostgreSQL + Redis + MinIO
```

## 6. 风险更新

| 风险 | V1.0评估 | V2.0变化 | 缓解措施 |
|------|----------|---------|----------|
| 判卷延迟 | MEDIUM | → HIGH(核心路径依赖) | GPU 独占 + 规则优先 + 批量处理 |

| 测试覆盖 | HIGH(无测试) | → 目标 ≥80% | TDD + pytest + Playwright |
| 数据迁移 | LOW | → MEDIUM(SQLite→PG) | Alembic 统一管理 |
| DGX 单点故障 | LOW | → LOW | 热备模型 + K8s 自动调度 |

## 7. 总结

V1.0 已构建完整的 Web 框架骨架(68个API + 17张表 + 20个前端页面), 验证了技术选型的可行性。V1.0 的核心短板在于**业务引擎空壳化** — 判卷、OCR、错题本、通知等核心业务逻辑未实质实现。

V2.0 聚焦于 **"打通四条核心链路"**:
1. **判卷链路**: 规则匹配引擎 → 自动判卷 → 结果反馈
2. **错题本链路**: 错题收集 → 知识点匹配 → 强化练习 → 文档导出
3. **OCR 链路**: MinIO → PaddleOCR → 结构化 → 人工校验
4. **通知链路**: WebSocket + SMTP 实时推送

同时偿还关键技术债务: PostgreSQL 迁移、测试覆盖、安全加固、部署自动化。

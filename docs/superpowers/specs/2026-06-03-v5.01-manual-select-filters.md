# V5.01 手工选题抽屉筛选条件增强

**日期**: 2026-06-03
**状态**: 已确认（二轮架构审阅通过）
**审阅**: architect agent, 首轮发现 3C+3H+3M，二轮审核旧知识点清理方案推荐分阶段执行
**范围**: Step 3 RecommendStep 手工选题抽屉

## 目标

手工选题抽屉当前只有关键词搜索，增加**题型**、**难度**、**知识点**三个筛选条件，并支持从当前单元/试卷自动填充初始值。

## UI 设计

### 布局：折叠筛选面板（方案 B）

- 搜索框右侧新增「筛选」按钮，点击展开/折叠筛选面板
- 因打开抽屉时已自动填充了值，**默认展开**让用户确认
- 筛选面板内三个控件横排：题型 Select、难度 Select、知识点 TreeSelect

### 筛选控件

| 控件 | 组件 | 模式 | 自动填充来源 | 空值默认 |
|------|------|------|-------------|---------|
| 题型 | `Select` | 多选 | `unit.question_config[].question_type` 去重 | 全部不选（等同不限题型） |
| 难度 | `Select` | 多选 | `paper.difficulty_ratio` 中比例 > 0 的 key | 全部选中（等同不限难度） |
| 知识点 | `TreeSelect` | 多选 | `paper.knowledge_node_ids`，加载最新版知识树 | 全部不选（等同不限知识点） |

选项枚举值：
- 题型：`SINGLE_CHOICE`, `MULTIPLE_CHOICE`, `FILL_BLANK`, `SUBJECTIVE`
- 难度：`EASY`, `MEDIUM`, `HARD`

### 交互行为

- 筛选条件变更后 **debounce 300ms** 自动重新搜索
- 关键词搜索与筛选条件是 **AND 关系**（叠加过滤）
- 搜索结果仍排除已在试卷中的题目
- 结果卡片补充显示**分数**、**知识点标签**（从 `meta_data.knowledge_points` 读取）

### 防御性检查

- 替换题目前验证 `manualTarget.unitId` 对应的 unit 仍存在于 `paper.units` 中（autoSave 刷新可能使引用失效）

## 知识点筛选方案（方案 C 阶段 1）

### 现实约束

经调查发现：
- `QuestionKnowledgeNode` 关联表**存在但完全为空**（无 seed 数据、无迁移填充）
- 所有题目知识点存储在 `meta_data.knowledge_points`（JSON 列）和 `grade_level.knowledge_points`（JSONB 列），值为**文本标签**
- **数据不均衡**: 手工创建题目只写 `meta_data.knowledge_points`，LLM/爬取题目同时写两个列

### 采用方案：后端 UUID→名称桥接 + 双列 OR 过滤

API 参数保持 `knowledge_point_ids`（UUID），**后端**负责将 UUID 转换为文本名称后做双列 OR 过滤。

```python
# GET /questions 新增参数
knowledge_point_ids: Optional[str] = None  # 逗号分隔的 KnowledgeNode UUID

# 查询逻辑
if knowledge_point_ids:
    kid_list = []
    for k in knowledge_point_ids.split(","):
        k = k.strip()
        if not k:
            continue
        try:
            kid_list.append(uuid.UUID(k))
        except ValueError:
            continue
    if kid_list:
        kn_result = await db.execute(
            select(KnowledgeNode.name).where(KnowledgeNode.id.in_(kid_list))
        )
        kp_names = [row[0] for row in kn_result.fetchall()]
        if kp_names:
            from sqlalchemy import or_
            query = query.where(or_(
                Question.grade_level['knowledge_points'].contains(kp_names),
                Question.meta_data.contains({"knowledge_points": kp_names}),
            ))
```

**设计要点**:
- 前端传 UUID，后端查 `KnowledgeNode.name` 做桥接，前端无需感知名称映射
- 双列 OR 覆盖 100% 题目来源（手工创建 + LLM/爬取）
- 零写路径改动，零数据迁移
- 名称匹配前做 `strip()` 统一空格规范化，降低假阴性

### 后续阶段（V5.x 独立立项）

`QuestionKnowledgeNode` 关联表的完整迁移（数据迁移 + 5+ 写路径改造 + 4+ 读路径切换到 JOIN）范围大、风险高（名称→UUID 匹配准确率不确定），独立立项执行。

## API 变更

### CRITICAL: question_type / difficulty 支持逗号分隔多值

**问题**: 当前 L638-641 使用 `==` 精确匹配，前端发送 `"SINGLE_CHOICE,MULTIPLE_CHOICE"` 会匹配不到任何记录。

**修复**: 改为 `in_()` 操作符。

```python
# question_type（L638-639）
if question_type:
    type_list = [t.strip() for t in question_type.split(",") if t.strip()]
    query = query.where(Question.question_type.in_(type_list))

# difficulty（L640-641）
if difficulty:
    diff_list = [d.strip() for d in difficulty.split(",") if d.strip()]
    query = query.where(Question.difficulty.in_(diff_list))
```

### 前端 API 调用

```typescript
const params: any = {
```

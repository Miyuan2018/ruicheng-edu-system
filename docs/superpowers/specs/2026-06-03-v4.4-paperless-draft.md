# V4.4 草稿去纸化 — 新建试卷 Step 4 前不产生主表记录

> **目标**：新建试卷在 Step 4「预览保存」之前，所有中间状态仅存草稿表（`exam_paper_drafts`，`paper_id=NULL`），不在 `exam_papers` 主表产生空壳记录。

## 动机

当前 `autoSave` 为了调用 `POST /{paper_id}/auto-generate` 和 `POST /{paper_id}/questions/{qid}/swap`，必须先在主表创建记录拿 `id`。导致用户刚填完基本信息，试卷就出现在列表中。若用户中途放弃，遗留空壳记录。

**核心洞察**：这两个端点实际只用 paper 的 `subject`、`grade_level`、`knowledge_node_ids` 三个字段，全部存在于前端 Zustand store 中，不依赖 paper 在数据库中存在。

## 架构

```
新建流程:
  Step 0~1 → autoSave → 草稿表 (paper_id=NULL)
  Step 2   → autoGenerate/swap → 新 paperless 端点 (不查 paper 表)
  Step 3   → saveAll → 创建主表记录 → saveAll → 删草稿
                         ↑ 此时才首次写入 exam_papers

编辑流程: 完全不受影响，paper.id 始终存在
```

## 后端改动

### 1. 新增 `POST /api/v1/exam-papers/auto-generate`（paperless）

无路径参数 `paper_id`。与现有 `POST /{paper_id}/auto-generate` 共享同一推荐引擎逻辑。

```
Request Body:
{
  subject: str,                        // 新增，替代从 DB paper 读取
  difficulty_ratio: {EASY, MEDIUM, HARD},
  knowledge_node_ids: [str],
  existing_question_ids: [str],
  type_configs: [{question_type, count, score_per_question}]
}

Response: {questions: [...], constraint_dashboard: {...}}
```

实现：复用 `distribute_quotas` + `select_for_targets`，`subject` 从 body 取。原有端点保留不动。

### 2. 新增 `POST /api/v1/questions/{question_id}/swap`（paperless）

脱离 paper 路径，注册到 questions router。

```
Request Body:
{
  subject: str | None,
  grade_level: {scope, grades} | None,
  knowledge_node_ids: [str] | None,
  exclude_ids: [str]                 // 已选题 ID，避免换回重复题
}

Response: {alternatives: [{question_id, title, difficulty, question_type, score}]}
```

实现：从现有 swap 逻辑提取，去掉 paper 查询，已用题目通过 `exclude_ids` 传入。

### 4. 草稿 upsert 适配 `paper_id=NULL`

`POST /api/v1/drafts` 中查询逻辑：

```python
if body.paper_id:
    existing = await db.execute(
        select(ExamPaperDraft).where(
            ExamPaperDraft.user_id == current_user.id,
            ExamPaperDraft.paper_id == body.paper_id,
        )
    )
else:
    # paper_id 为 null 时，按 user_id 查最近一条无关联草稿
    existing = await db.execute(
        select(ExamPaperDraft).where(
            ExamPaperDraft.user_id == current_user.id,
            ExamPaperDraft.paper_id.is_(None),
        ).order_by(ExamPaperDraft.updated_at.desc()).limit(1)
    )
draft = existing.scalar()  # 用 scalar() 而非 scalar_one_or_none()，避免多行异常
```

        grade_level: gl,

      });
      pid = resp.data?.id || resp.data;
      set({ paper: { ...get().paper, id: pid } });
    }

    // 阶段 2：写主表
    const cleanPaper = { /* 构建 cleanPaper 不变 */ };
    await paperApi.saveAll(pid, cleanPaper);

    // 阶段 3：按 draftId 精确删除草稿
    if (draftId) {
      await draftApi.delete(draftId).catch(() => {});
    } else {
      // 编辑流程兜底：按 paper_id 查草稿并删除

  } catch (e) {
    set({ saving: false });
    throw e;
  }

## 不受影响的范围

- **PaperListPage**：列表查询不依赖草稿表，空壳消失后列表自然干净
- **学生端答题/提交**：操作已发布/已保存的试卷，paper 必然存在
- **发布/复制/导出**：仅在已保存的试卷上操作
- **编辑流程**：paper.id 始终存在，所有现有逻辑不变
- **StructureStep**：不引用 paper.id，仅操作 paper.units

## 已知限制

- **多标签页新建**：`paper_id=NULL` 的草稿在 PostgreSQL 唯一约束中视为互异，同一用户可有多个 null 草稿。upsert 用 `LIMIT 1` 取最近一条，多标签页共享同一草稿。此场景极为罕见，不做额外处理。

## 端到端流程验证

### 新建流程
1. 用户点击「新建试卷」→ initNew() → paper.id=null
2. 填写基本信息 → 下一步 → autoSave → 草稿表 (paper_id=null)
3. 配置试卷结构 → 下一步 → autoSave → 草稿更新
4. 进入选题步骤 → useEffect 触发 → regenerateAll → 调 paperless auto-generate ✓
5. 换题 → handleSwap → 调 paperless swap ✓
6. 进入预览保存 → 点击保存 → saveAll 先建主表 → saveAll → 删草稿 ✓
7. 主表出现完整试卷记录，草稿表清理完毕

### 编辑流程
1. 从列表点击编辑 → loadDraft(id) → paper.id 存在
2. 所有步骤与现有逻辑一致，不受影响

### 中断恢复
1. 新建中途关闭浏览器 → 草稿 (paper_id=null) 留存
2. 再次点击「新建试卷」→ pendingDraft 检出 → 恢复弹窗
3. 用户选择继续 → 恢复草稿状态 → 可继续编辑

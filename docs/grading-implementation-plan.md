# edu_system 判卷服务实现计划

> **V1.0 状态**: ⚠️ API端点存在(9个), ❌ 规则匹配/LLM语义评分引擎未实现。
> V2.0 S1: 规则匹配引擎(客观题) → LLM语义评分(主观题, 二期)。
> 详见 `docs/requirements-v2.0.md`

## 1. 判卷服务概述

判卷服务是edu_system的核心组件，负责自动评分学生的答案。根据系统架构和关键数据流描述，判卷服务的工作流程为：
前端 → 答案服务 → （临时存储） → 判卷服务 → 结果 → 错题本服务（如果有错题）

判卷服务需要处理两种主要类型的题目：
1. **客观题**（选择题、填空题）：使用规则匹配进行精确判分
2. **主观题**（解答题、作文题）：使用大模型进行语义评分

## 2. 技术栈选择

### 2.1 判卷引擎
根据CLAUDE.md和需求文档，判卷服务采用混合策略：
- **规则匹配引擎**：处理选择题和填空题，基于预定义规则进行精确比较
- **大模型语义评分引擎**：处理解答题和作文题，使用LLM进行语义理解和评分

### 2.2 服务架构
判卷服务采用微服务架构，主要组成部分：
- **API接口层**：基于FastAPI提供RESTful接口
- **任务队列处理器**：消费消息队列中的判卷任务
- **规则匹配处理器**：处理客观题的判卷
- **语义评分处理器**：调用LLM进行主观题的语义评分
- **结果聚合器**：合并各题目的判卷结果，计算总分
- **模型接口**：与模型管理服务集成获取可用的判卷模型
- **存储接口**：与数据库和对象存储交互
- **通知接口**：发送判卷完成通知

## 3. 功能设计

### 3.1 核心功能
1. **判卷任务接收**：接收来自答案服务的判卷请求
2. **题目类型路由**：根据题目类型分发到相应的判卷处理器
3. **客观题判卷**：使用规则匹配进行精确对比
4. **主观题判卷**：调用大模型进行语义理解和评分
5. **结果聚合**：合并所有题目的得分和反馈
6. **状态跟踪**：提供判卷状态查询接口
7. **异常处理**：处理判卷过程中的各种错误情况
8. **模型管理**：支持判卷模型的查询、切换和版本管理
9. **批量判卷**：支持批量试卷的判卷处理
10. **性能优化**：利用GPU加速和并发处理提高判卷速度

### 3.2 API接口设计（基于已完成的后端API设计计划）

#### 3.2.1 判卷触发
- **POST /api/v1/grading/start**
  - 功能：触发答案判卷
  - 请求体：{answer_submission_id: uuid, ...}
  - 响应：{grading_id: uuid, status: "PENDING", message: "Grading task queued"}
  
- **GET /api/v1/grading/status/{grading_id}**
  - 功能：查询判卷状态
  - 响应：{grading_id: uuid, status: "PENDING|PROCESSING|COMPLETED|FAILED", progress: int, ...}
  
- **GET /api/v1/grading/result/{grading_id}**
  - 功能：获取判卷结果
  - 响应：{grading_id: uuid, status: "COMPLETED", results: {...}, ...}

#### 3.2.2 判卷历史
- **GET /api/v1/grading/history/student/{student_id}**
  - 功能：获取学生判卷历史
  - 响应：[{grading_id: uuid, exam_paper_id: uuid, score: decimal, ...}, ...]
  
- **GET /api/v1/grading/history/exam/{exam_paper_id}**
  - 功能：获取试卷判卷统计
  - 响应：{exam_paper_id: uuid, total_submissions: int, avg_score: decimal, score_distribution: {...}, ...}

#### 3.2.3 模型管理
- **GET /api/v1/grading/models**
  - 功能：获取可用判卷模型列表
  - 响应：[{model_id: uuid, name: string, version: string, type: string, ...}, ...]
  
- **POST /api/v1/grading/models/switch**
  - 功能：切换判卷模型（管理员）
  - 请求体：{model_id: uuid}
  - 响应：{success: boolean, message: string}
  
- **GET /api/v1/grading/models/current**
  - 功能：获取当前使用的判卷模型
  - 响应：{model_id: uuid, name: string, version: string, ...}

## 4. 系统集成设计

### 4.1 答案服务集成
- 答案服务在答案提交完成后发送判卷任务到消息队列
- 判卷服务消费判卷任务队列
- 判卷完成后，将结果更新到答案提交记录中（通过答案服务API或直接数据库操作）
- 或者判卷服务直接调用答案服务的答案更新接口

### 4.2 消息队列集成 (RabbitMQ/Kafka)
- 判卷服务消费判卷任务队列
- 任务包含：答案提交ID、学生ID、试卷ID等信息
- 判卷结果可以通过回调队列返回，或者判卷服务直接更新数据库
- 失败任务进入死信队列或根据重试策略重新入队

### 4.3 模型管理服务集成
- 判卷服务从模型管理服务获取可用的判卷模型列表
- 判卷服务可以切换当前使用的判卷模型
- 判卷服务将使用的模型信息记录到判卷记录中
- 模型管理服务可以提供模型的性能指标用于判卷服务的自我优化

### 4.4 错题本服务集成
- 判卷服务不直接与错题本服务交互
- 判卷结果通过答案服务传递给错题本服务
- 错题本服务根据判卷结果识别错题并生成错题本

### 4.5 对象存储集成 (MinIO)
- 判卷服务可能需要存储中间结果或大的反馈内容
- 主观题的原始答案图片（如果是手写）可能存储在MinIO
- 判卷服务从MinIO获取需要评分的内容

## 5. 数据模型

基于之前设计的数据库架构，判卷服务主要涉及以下表：

### 5.1 判卷记录表 (grading_records)
在database-design.md中已经定义，判卷服务主要操作这个表：
- 创建记录（状态为PENDING）
- 更新判卷状态和进度
- 存储判卷结果（details字段包含详细结果）
- 记录使用的模型信息（model_used, model_version）
- 记录错误信息（如果失败）
- 更新时间戳

### 5.2 关联表
- 通过answer_submission_id关联答案提交表
- 间接关联学生信息（通过答案提交表）
- 间接关联试卷信息（通过答案提交表）

## 6. 判卷算法设计

### 6.1 客观题判卷（选择题、填空题）
对于客观题，判卷服务使用精确规则匹配：

#### 6.1.1 选择题判卷
1. 获取题目的标准答案（从questions表的correct_answer字段）
2. 获取学生的答案（从answer_details表的student_answer字段）
3. 进行标准化处理（去除空格、统一大小写等，根据题目要求）
4. 精确比较：如果学生答案等于标准答案，则判为正确
5. 计分：如果正确，获得题目满分；否则得0分
6. 反馈：提供正确答案和学生答案的对比

#### 6.1.2 填空题判卷
1. 获取题目的标准答案（可能是一个或多个空的答案）
2. 获取学生的答案（对应每个空的答案）
3. 根据题目要求进行标准化处理
4. 逐空比较：每个空的答案分别判断对错
5. 计分：根据每空的分值计算总分（可以设置部分得分）
6. 反馈：标注哪些空答对、哪些空答错，并给出正确答案

### 6.2 主观题判卷（解答题、作文题）
对于主观题，判卷服务使用大模型进行语义评分：

#### 6.2.1 预处理阶段
1. 获取题目信息和标准答案或评分标准
2. 获取学生答案
3. 文本预处理：去除多余空格、统一格式等
4. 如果答案是图片（手写），先进行OCR识别（可能调用OCR服务）

#### 6.2.2 提示词工程
根据不同的题目类型和评分要求，构建合适的提示词促使LLM进行公平评分：

**解答题提示词模板：**
```
你是一位经验丰富的老师，需要评分以下解答题。

题目：[题目内容]
题型：解答题
总分：[题目分值]
评分标准：[如果有提供的评分标准]

学生答案：
[学生的答案]

请按照以下要求进行评分：
1. 首先判断答案是否正确解答了问题
2. 评估答案的完整性（是否覆盖了所有关键点）
3. 评估答案的准确性（概念是否正确，推理是否合理）
4. 评估答案的表达是否清晰逻辑
5. 根据以上方面给出0-[总分]之间的分数
6. 提供简洁的评语说明得分理由

请以JSON格式返回结果：
{
  "score": 分数值,
  "feedback": "评语内容",
  "key_points_covered": [覆盖的关键点列表],
  "missing_points": [遗漏的关键点列表],
  "accuracy": 准确性评估,
  "completeness": 完整性评估,
  "clarity": 表达清晰度评估
}
```

**作文题提示词模板：**
```
你是一位资深语文老师，需要评分以下作文。

题目：[作文题目]
题型：作文
总分：[题目分值]
作文类型：[记叙文/议论文/说明文等]
评分维则：[如果有提供的评分标准，如：思想内容、结构、语言、字数等]

学生作文：
[学生的作文内容]

请按照以下要求进行评分：
1. 思想内容：观点是否正确健康，是否紧扣主题
2. 结构安排：结构是否完整合理，是否有明确的开头、主体和结尾
3. 语言表达：语言是否流畅准确，是否有修辞手法运用
4. 字数要求：是否符合字数要求
5. 根据以上维度给出0-[总分]之间的分数
6. 提供详细的评语和改进建议

请以JSON格式返回结果：
{
  "score": 分数值,
  "feedback": "详细评语",
  "ideas_score": 思想内容得分,
  "structure_score": 结构得分,
  "language_score": 语言得分,
  "word_count": 实际字数,
  "suggestions": [改进建议列表]
}
```

#### 6.2.3 模型调用和结果解析
1. 调用选定的判卷模型（通过模型管理服务获取当前模型信息）
2. 将构建好的提示词发送给LLM
3. 接收LLM返回的结果
4. 解析JSON结果，提取得分和反馈
5. 验证结果的合理性（分数在有效范围内）
6. 如果需要，可以进行多次采样取平均值以提高一致性（尤其对于边界情况）

#### 6.2.4 质量控制
- 设置置信度阈值：如果模型输出的置信度过低，可以标记为需要人工复核
- 一致性检查：对同一答案进行多次判卷，检查得分方差
- 异常检测：检测是否有异常高或低的得分
- 人工复核流程：对于争议大或置信度低的答案，标记为需要人工判卷

## 7. 性能优化

### 7.1 GPU资源管理
- 根据CLAUDE.md建议：判断和OCR服务的GPU资源隔离
- 使用容器编排（Kubernetes）实现GPU资源的专用分配
- 设置判卷服务pod的GPU限制和请求
- 实现GPU内存监控和自动扩容

### 7.2 并发处理
- 判卷服务支持多工作进程并行处理队列任务
- 主观题判卷可以利用LLM的批量推理能力（一次发送多个 prompt）
- 客观题判卷由于计算量小，可以实现非常高的吞吐量
- 合理配置队列消费者数量避免资源竞争

### 7.3 规则匹配优化
- 预编译常用的规则模式
- 使用哈希表快速查找标准答案
- 对于选择题，可以预计算所有可能答案的哈希值
- 批量处理客观题时利用向量化操作

### 7.4 语义评分优化
- 提示词缓存：对于相同的题目和评分标准，可以缓存提示词模板
- 批量推理：一次发送多个判卷请求给LLM（如果LLM支持）
- 模型预热：保持模型在内存中以减少加载时间
- 结果缓存：对于完全相同的答案（罕见但可能），可以缓存判卷结果
- 异步处理：判卷服务可以异步返回结果，不阻塞API请求

### 7.5 数据库优化
- 判卷记录表适当索引（已在数据库设计中考虑）
- 考虑将详细结果（details字段）中较大的部分存储在对象存储中
- 批量更新判卷结果以减少数据库写入次数

## 8. 安全考虑

### 8.1 数据安全
- 判卷过程中的学生答案和试题内容需要保密
- 判卷服务仅具有读取必要数据和写入判卷结果的权限
- 使用HTTPS/TLS加密所有服务间通信
- 判卷模型本身可能包含敏感信息，确保模型存储和传输安全

### 8.2 服务安全
- 判卷API接口要求JWT token认证（除内部服务间通信外）
- 实施速率限制防止恶意判卷请求
- 输入验证：验证答案提交ID等参数的有效性
- 防止重复判卷：同一答案提交在特定时间内只能触发一次判卷（除非显式重新判卷）

### 8.3 模型安全
- 判卷模型的来源需要可信任
- 模型输入输出需要过滤以防止提示词注入攻击
- 定期更新和修补判卷模型以修复已知漏洞
- 模型使用过程监控以检测异常行为

## 9. 配置管理

判卷服务需要以下配置：
- 判卷模型选择和参数（默认模型、备用模型列表等）
- 规则匹配参数（选择题/填空题的匹配严格程度等）
- 语义评分参数（温度参数、最大token数、重试次数等）
- MinIO连接信息（如果需要存储中间结果）
- 消息队列连接信息
- 数据库连接信息
- 服务端口和地址
- 日志级别和输出方式
- 重试策略（最大重试次数、重试间隔）
- 超时设置（判卷处理超时、模型调用超时）
- 批处理大小限制
- 资源限制（内存、CPU、GPU使用率警告阈值）
- 质量控制参数（置信度阈值、一致性检查参数等）
- 人工复核触发条件

配置应通过环境变量、配置文件或模型管理服务提供，支持热更新。

## 10. 错误处理和监控

### 10.1 错误类型
- 答案数据获取失败（数据库查询、关联信息缺失）
- 题目信息不完整（缺少正确答案或评分标准）
- 规则匹配异常（正则表达式错误等）
- 模型调用失败（服务不可超时、返回格式错误）
- 结果解析失败（LLM返回非JSON或字段缺失）
- 数据库写入失败
- 队列通信失败

### 10.2 错误处理策略
- 错误分类：可重试错误 vs 不可重试错误
- 指数退避重试策略（尤其对于模型调用失败）
- 错误通知：发送失败通知给答案服务或前端
- 人工干预：对于反复失败的判卷，标记为需要人工处理
- 降级策略：某些情况下可以尝试使用备用判卷模型或简化评分策略
- 熔断机制：如果判卷服务自身出现问题，暂停接受新任务

### 10.3 监控指标
- 判卷处理速率（任务/分钟）
- 平均判卷时间（客观题 vs 主观题）
- 成功率和失败率
- 各种错误类型分布
- 客观题判卷准确率（如果有标准答案可验证）
- 主观题判卷一致性（多次判卷的得分方差）
- 队列长度和等待时间
- 资源使用率（CPU、内存、GPU、磁盘IO）
- API响应时间和错误率
- 模型调用成功率和平均响应时间
- 分数分布统计（判卷结果的分数分布）

### 10.4 日志策略
- 结构化日志便于日志系统解析
- 不同级别日志（DEBUG, INFO, WARN, ERROR）
- 关键操作日志（判卷开始、完成、失败）
- 性能日志（判卷时间分布、模型调用时间）
- 错误日志包含必要的上下文信息（但避免记录完整学生答案）
- 判卷结果日志（仅用于调试，生产环境应谨慎）

## 11. 部署和扩展策略

### 11.1 容器化部署
- 使用Docker容器封装判卷服务
- 容器镜像包含判卷逻辑和必要依赖（但不包含大模型，模型通过模型管理服务获取）
- 通过容器编排平台（Kubernetes）管理服务实例

### 11.2 扩展策略
- 水平扩展：根据队列长度增加判卷服务实例数量
- 垂直扩展：根据监控调整单实例资源分配
- 动态扩展：基于队列深度和处理时间的自动伸缩策略
- GPU自动伸缩：在支持的平台上根据GPU使用率调整（用于语义评分）
- 客观题判卷和主观题判卷可以部署为不同的服务实例，根据需求独立扩展

### 11.3 发布策略
- 蓝绿发布或滚动更新确保服务可用性
- 数据库向后兼容：新版本服务能够处理旧版本判卷记录格式
- 配置版本控制：跟踪配置变更便于回滚
- 功能标志：逐步推出新判卷策略而不影响现有用户
- 模型版本发布：通过模型管理服务安全地切换判卷模型

- 测试与消息队列的集成（发送、消费、确认）
- 测试与模型管理服务的集成（模型查询和切换）
- 测试端到端流程：答案提交→判卷→结果返回
- 测试不同题型的判卷流程
- 测试批量判卷处理
- 测试错误恢复和重试机制
- 测试人工复核触发条件

### 12.3 性能测试
- 并发处理测试：模拟多个同时提交判卷的场景
- 大卷批处理测试：测试大量试卷的判卷能力
- 主观题判卷性能测试：测试不同长度答案的处理时间
- 资源消耗测试：监控不同负载下的资源使用
- 延迟测试：测试从答案提交到判卷结果返回的端到端延迟
- 压力测试：测试系统在峰值负载下的表现

### 12.4 准确性测试
- 客观题准确性测试：使用标准答案集测试规则匹配的准确率
- 主观题一致性测试：评估不同判卷对同一答案的一致性
- 基准比较：如果可能，与人工判卷结果进行比较
- 边界情况测试：测试各种特殊答案（空答案、极长答案、非法字符等）
- 公平性测试：评估判卷是否对不同写法但含义相同的答案给予相似得分

### 12.5 安全测试
- 输入验证测试：测试各种恶意输入的处理
- 认证和授权测试：确保未授权访问被正确拒绝
- 数据泄露测试：检查日志和错误信息中是否泄露敏感数据
- 模型安全测试：测试是否存在提示词注入漏洞
- 并发安全测试：测试高并发下的数据一致性

## 14. 验收标准

1. 所有API接口按照设计实现并通过功能测试
2. 客观题判卷准确率达到预期目标（>99%）
3. 主观题判卷一致性达到预期目标（同一答案多次判卷得分方差<0.5分）
4. 平均判卷时间符合性能要求（简单客观题<0.5秒，复杂主观题<5秒）
5. 服务能够水平扩展以处理峰值负载
6. 错误处理机制完善，能够优雅地处理各种失败情况
7. 安全扫描未发现高危漏洞
8. 监控和告警系统能够及时发现和报告问题
9. 模型管理功能正常，能够切换不同的判卷模型
10. 部署文档清晰，能够在不同环境中成功部署
11. 性能基准测试报告符合预期目标
12. 准确性测试报告显示判卷质量满足教学要求

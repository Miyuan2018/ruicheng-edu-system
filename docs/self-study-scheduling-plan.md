# edu_system 自学调度服务规划

> **V1.0 状态**: ❌ API端点占位(15个), 核心逻辑全部未实现。属于二期功能。
> 详见 `docs/requirements-v2.0.md`

## 1. 自学调度服务概述

自学调度服务是edu_system的智能核心组件，负责驱动系统的“测验 → 整理错题 → 订正 → 加深训练”闭环循环。根据系统架构和关键数据流描述，自学调度服务的工作流程为：
调度服务 → 网络爬虫 → 知识点建模 → 题目生成 → 模型微调 → 模型注册表 → 判卷服务（更新模型）

自学调度服务不仅是一个简单的任务调度器，更是一个智能学习系统的“大脑”，能够根据学生的学习情况自动生成个性化学习内容，持续优化教学效果。

## 2. 技术栈选择

根据CLAUDE.md和需求文档，自学调度服务将使用：
- **工作流编排**：Apache Airflow 或 Prefect（如CLAUDE.md所述）
- **后端服务**：Python 3.11+ + FastAPI（提供API接口）
- **网络爬虫**：Scrapy、BeautifulSoup 或 Selenium
- **知识点建模**：自然语言处理技术（NLTK、spaCy、transformers等）
- **题目生成**：基于模板的生成或大模型辅助生成
- **模型微调**：HuggingFace Transformers + PEFT (LoRA/QLoRA)
- **任务队列**：RabbitMQ 或 Kafka（用于细粒度任务调度）
- **数据存储**：PostgreSQL + SQLAlchemy ORM（已有数据库设计）
- **模型存储**：MinIO 对象存储（用于模型文件）
- **模型注册表**：MLflow 或 DVC（如CLAUDE.md所述）

## 3. 功能设计

### 3.1 核心功能
1. **自学任务管理**：创建、查询、更新、删除自学任务
2. **知识点建模**：从抓取的网络内容中自动提取和建模知识点
3. **题目生成**：基于知识点自动生成练习题
4. **模型微调触发**：触发和监控模型微调训练过程
5. **数据同步**：触发题库数据同步操作
6. **智能触发机制**：基于学习数据自动触发相应的自学任务
7. **任务监控和告警**：实时监控任务状态并发送告警
8. **工作流可视化**：提供任务执行的可视化界面
9. **依赖管理**：处理任务之间的依赖关系
10. **失败恢复**：支持任务失败时的重试和回滚机制

## 4. 系统集成设计

### 4.1 题目服务集成
- 自学调度服务从题目服务获取现有题目信息，用于知识点建模和题目生成的参考
- 题目生成功能可以调用题目服务的接口验证生成题目的质量和唯一性
- 知识点建模结果可以反馈到题目服务，帮助题目的分类和标签

### 4.2 判卷服务集成
- 自学调度服务可以接收判卷服务的结果，用于分析学生学习情况
- 模型微调触发功能可以生成专门用于判卷的模型
- 生成的判卷模型可以注册到模型管理服务，供判卷服务使用

### 4.3 错题本服务集成
- 自学调度服务可以从错题本服务获取学生错题分析结果
- 错题本的知识点分析可以作为知识点建模和题目生成的输入
- 自学任务可以基于错题本结果自动生成（例如：为薄弱知识点创建练习任务）

### 4.4 模型管理服务集成
- 自学调度服务是模型管理服务的主要消费者
- 自学调度服务触发的模型微调会生成新模型版本
- 新模型版本会注册到模型管理服务
- 其他服务（如判卷服务、题目生成服务）可以从模型管理服务获取最新模型

### 4.5 网络爬虫集成
- 知识点建模功能依赖于网络爬虫获取内容
- 可以内置爬虫或调用外部爬虫服务
- 爬虫结果作为知识点提取的输入
- 爬虫可以定时运行或基于特定触发条件运行

### 4.6 对象存储集成 (MinIO)
- 模型微调产生的模型文件存储到MinIO
- 题目生成的中间结果或模板可以存储到MinIO
- 知识点建模的原始内容可以临时存储到MinIO
- 数据同步可能涉及从MinIO读取或写入数据

### 4.7 消息队列集成 (RabbitMQ/Kafka)
- 自学调度服务使用消息队列进行任务解耦和异步处理
- 任务创建后发送到相应的队列，由工作进程消费
- 任务状态更新可以通过回调队列或直接数据库更新
- 失败任务可以进入死信队列进行重试或人工处理

## 5. 工作流编排设计

自学调度服务的核心是工作流编排，基于CLAUDE.md提到的Airflow/Prefect。

### 5.1 工作流类型
1. **知识点建模工作流**：爬取内容 → 内容预处理 → 知识点提取 → 知识点验证 → 知识点存储
2. **题目生成工作流**：知识点选择 → 题目模板选择/提示词构建 → 题目生成 → 题目审核 → 题目存储
3. **模型微调工作流**：数据准备 → 数据预处理 → 模型加载 → 训练过程监控 → 模型评估 → 模型保存 → 模型注册
4. **数据同步工作流**：数据源连接 → 数据提取 → 数据清洗 → 数据转换 → 数据加载 → 验证和更新
5. **智能学习闭环工作流**：答案分析 → 错题识别 → 知识点薄弱点分析 → 题目生成 → 模型更新 → 再次测验

### 5.2 工作流组件
- **有向无环图(DAG)**：定义任务之间的依赖关系
- **操作符(Operators)**：具体执行的任务单元（如：爬虫操作符、知识点提取操作符）
- **触发器(Triggers)**：什么情况下启动工作流（如：定时触发、事件触发、手动触发）
- **调度器(Scheduler)**：决定什么时候运行什么任务
- **执行器(Executor)**：实际执行任务的工作进程
- **元数据库**：存储工作流状态、任务历史等信息
- **用户界面**：监控和管理工作流的Web界面

### 5.3 任务类型详细设计

#### 5.3.1 知识点提取任务
- **输入**：网络内容（URL或直接文本）、科目、年级
- **过程**：
  1. 内容获取（如果是URL则爬取）
  2. 内容预处理（去除HTML标签、正则文本等）
  3. 知识点识别（使用规则匹配、NLP或LLM）
  4. 知识点标准化（合并重复、建立层级关系）
  5. 知识点验证（检查准确性和完整性）
  6. 知识点存储（保存到知识点表）
- **输出**：提取的知识点列表及其属性
- **依赖**：可能依赖于内容获取任务
- **后续**：可以触发题目生成任务或知识点验证任务

#### 5.3.2 题目生成任务
- **输入**：知识点ID列表、题型、难度、数量
- **过程**：
  1. 题目模板选择或提示词构建
  2. 调用题目生成引擎（模板-based 或 LLM-based）
  3. 生成的题目初步审核（格式、重复性等）
  4. 必要时进行人工审核（对于关键题目）
  5. 存储生成的题目到题目服务（如果需要）
  6. 生成题目的元数据和关联关系
- **输出**：生成的题目列表及其详情
- **依赖**：知识点必须存在
- **后续**：可以触发模型训练任务（使用新题目作为训练数据）或直接用于练习

#### 5.3.3 模型微调任务
- **输入**：基础模型、训练数据、训练配置
- **过程**：
  1. 数据准备和预处理
  2. 模型加载和初始化
  3. 训练过程监控（损失、准确率等）
  4. 验证集评估
  5. 模型保存和格式转换
  6. 模型注册到模型管理服务
- **输出**：训练好的模型及其性能指标
- **依赖**：训练数据必须可用，基础模型必须存在
- **后续**：可以触发判卷服务模型更新或题目生成服务优化

#### 5.3.4 数据同步任务
- **输入**：数据源类型和同步参数
- **过程**：
  1. 连接数据源
  2. 提取需要同步的数据
  3. 数据清洗和格式转换
  4. 数据验证和质量检查
  5. 数据加载到目标系统
  6. 同步结果报告生成
- **输出**：同步的记录数量和状态
- **依赖**：数据源必须可访问
- **后续**：可以触发依赖于新数据的其他任务（如知识点建模或题目生成）

### 5.4 工作流触发机制
1. **定时触发**：基于cron表达式的定时工作流（如：每日知识点更新）
2. **事件触发**：响应特定事件启动工作流（如：判卷完成后触发错题分析）
3. **手动触发**：用户通过API或界面手动启动工作流
4. **依赖触发**：前置任务完成后自动触发后续任务
5. **条件触发**：基于特定条件（如：错误率超过阈值）触发工作流

### 5.5 工作流监控和管理
- 实时任务状态监控
- 任务执行时间和资源使用统计
- 失败任务自动重试机制
- 工作流可视化编辑界面
- 任务日志和审计 trail
- 性能瓶颈分析和优化建议
- 资源调度和负载均衡

## 6. 数据模型

### 6.1 自学任务表 (self_study_tasks)
在database-design.md中已经定义，自学调度服务主要操作这个表：
- 创建记录（状态为PENDING）
- 存储任务基本信息（类型、参数、优先级等）
- 更新任务状态和进度
- 记录任务依赖关系
- 存储任务结果和错误信息
- 管理任务的生命周期

### 6.2 知识点建模表 (knowledge_point_models)
在database-design.md中已经定义，自学调度服务主要操作这个表：
- 存储从抓取内容中提取的知识点信息
- 关联来源内容和提取方法
- 记录提取的知识点列表和置信度
- 支持知识点的版本追溯和来源追踪

### 6.3 模型管理表 (ml_models)
在database-design.md中已经定义，自学调度服务主要操作这个表：
- 注册新生成的模型版本
- 更新模型的性能指标和状态
- 管理模型的生命周期（激活、废弃等）
- 支持模型的回滚和版本比较

### 6.4 其他关联表

## 7. 智能学习闭环设计

自学调度服务的终目标是实现智能学习闭环，即CLAUDE.md所述的“自学循环”：
调度服务 → 网络爬虫 → 知识点建模 → 题目生成 → 模型微调 → 模型注册表 → 判卷服务（更新模型）

### 7.1 闭环触发条件
1. **定时触发**：每日/每周自动执行一次完整闭环
2. **阈值触发**：当学生错误率超过某个阈值时触发
3. **里程碑触发**：完成某个学习阶段或题目数量后触发
4. **手动触发**：老师或系统管理员手动启动
5. **模型过期触发**：当前模型使用时间过长或性能下降时触发

### 7.2 闭环工作流详细设计

#### 7.2.1 阶段1：答案分析和错题识别
- **输入**：最近一段时间的答案提交和判卷结果
- **过程**：
  1. 查询学生答题记录（通过答案服务）
  2. 识别错误答案（基于判卷结果）
  3. 按知识点、题型、难度等维度分析错误
  4. 生成错误报告和薄弱点识别
- **输出**：学生知识点掌握情况报告
- **后续**：触发知识点建模或题目生成任务

#### 7.2.2 阶段2：知识点更新和增强
- **输入**：错题分析结果、现有知识点体系
- **过程**：
  1. 根据错题分析确定需要重点更新的知识点
  2. 爬取相关的最新网络内容
  3. 从网络内容中提取更新的知识点
  4. 与现有知识点体系进行合并和更新
  5. 验证新知识点的准确性和完整性
- **输出**：更新后的知识点体系
- **后续**：触发题目生成任务

#### 7.2.3 阶段3：个性化题目生成
- **输入**：更新的知识点体系、学生错题分析
- **过程**：
  1. 根据学生薄弱知识点选择目标知识点
  2. 基于目标知识点生成针对性练习题
  3. 可以根据学生历史表现调整题目难度
  4. 审核生成题目的质量和适用性
- **输出**：个性化练习题集
- **后续**：可以直接用于学生练习或触发模型训练

#### 7.2.4 阶段5：模型优化和更新
- **输入**：学生答题数据、生成的练习题、教学反馈
- **过程**：
  1. 准备模型训练数据（学生答案、正确答案、反馈等）
  2. 触发判卷模型的微调训练
  3. 监控训练过程并评估模型性能
  4. 将优化后的模型注册到模型管理服务
  5. 通知判卷服务使用新模型（或自动切换）
- **输出**：优化后的判卷模型
- **后续**：新模型用于后续判卷，完成闭环

#### 7.2.5 阶段6：效果评估和反馈
- **输入**：新模型的判卷结果、学生后续答题表现
- **过程**：
  1. 监控新模型的判卷准确率和一致性
  2. 分析学生在新模型下的学习进步
  3. 生成闭环效果报告
  4. 为下一轮闭环提供改进建议
- **输出**：闭环效果评估报告
- **后续**：根据评估结果调整闭环参数或触发下一轮

## 8. 性能和可靠性设计

### 8.1 任务调度优化
- 优先级调度：高优先级任务（如：错误修复）优先执行
- 资源感知调度：根据可用资源动态调整任务并发度
- 预测性调度：基于历史数据预测任务执行时间
- 死任务检测：自动检测和处理卡住的任务
- 任务分割：大任务拆分为小块以提高并发性和故障恢复能力

### 8.2 容错机制
- 重试策略：指数退避重试，可配置的最大重试次数
- 检查点机制：长任务设置检查点以便从中断处恢复
- 数据备份：关键中间结果定期备份
- 降级服务：某些功能失败时可以提供简化版服务
- 熔断机制：防止连锁故障，暂停对问题服务的调用

### 8.3 可扩展性设计
- 水平扩展：根据任务队列长度增加工作进程数量
- 功能隔离：不同类型的任务可以部署在不同的服务实例上
- 数据分区：大数据集可以按照时间或其他维度分区
- 缓存策略：频繁访问的数据使用缓存减少重复计算
- 数据库连接池：优化数据库连接使用以支持高并发

### 8.4 监控和告警
- 实时监控：任务执行速度、成功率、资源使用情况
- 自定义告警：任务失败、延迟过高、资源耗尽等情况
- 性能基准：定期运行性能测试建立基准线
- 趋势分析：分析任务执行时间的趋势以预防问题
- 日志聚合：集中收集和分析所有服务的日志

## 9. 安全考虑

### 9.1 数据安全
- 自学任务可能包含敏感的教育数据和学习行为
- 实施严格的访问控制：只有授权用户能够创建和修改任务
- 任务数据在传输和存储过程中使用HTTPS/TLS加密
- 模型和知识点数据需要防篡改保护
- 爬虫获取的内容需要审核以确保不包含恶意或不适当内容

### 9.2 服务安全
- 自学调度服务API接口要求JWT token认证
- 实施速率限制防止恶意请求（如：爬虫滥用）
- 输入验证：验证所有参数的合法性和安全性
- 防止越权访问：确保用户只能操作他们有权限的任务
- 安全的任务执行：限制任务可以执行的操作和访问的资源
- 审计日志：记录所有任务创建、修改和执行的行为

### 9.3 爬虫安全
- 遵守目标网站的robots.txt规则和服务条款
- 实施访问频率限制防止对目标网站造成压力
- 使用合适的User-Agent标识自身身份
- 处理获取内容时的法律和版权问题
- 考虑使用官方API而非爬虫（如果可用）

### 9.4 模型安全
- 模型训练数据需要脱敏处理，特别是学生个人信息
- 模型输出需要过滤以防止生成有害或不适当内容
- 模型训练过程需要监控以检测异常行为
- 模型存储和传输需要加密保护
- 模型使用过程需要监控以检测是否被用于恶意目的

## 10. 配置管理

自学调度服务需要以下配置：
- 工作流编排参数（调度间隔、重试策略、超时时间等）
- 爬虫参数（并发数、延迟、重试、超时、用户代理等）
- 知识点建模参数（提取方法、置信度阈值、验证规则等）
- 题目生成参数（模板库位置、LLM参数、质量控制标准等）
- 模型微调参数（默认训练配置、资源限制、PEFT选项等）
- 数据同步参数（同步频率、数据源配置、冲突处理策略等）
- 任务队列配置（连接信息、队列名称、死信队列策略等）
- 模型存储配置（MinIO连接、存储桶命名策略、清理策略等）
- 通知设置（默认渠道、模板、频率限制等）
- 任务优先级规则（不同类型任务的默认优先级）
- 资源限制（内存、CPU、GPU、磁盘IO使用率阈值）
- 日志级别和输出方式
- 依赖服务超时和重试策略
- 版本控制和回滚策略
- 国际化和本地化设置（语言、时区等）

配置应通过环境变量、配置文件提供，支持热更新。

## 11. 错误处理和监控

### 11.1 错误类型
- 任务创建失败（参数无效、权限不足等）
- 爬虫失败（网络问题、目标网站变动、反爬虫措施等）
- 知识点提取失败（内容无法解析、提取算法错误等）
- 题目生成失败（模板错误、LLM服务不可用等）
- 模型微调失败（数据问题、训练发散、资源不足等）
- 数据同步失败（数据源不可达、格式不匹配等）
- 工作流调度失败（依赖循环、资源不足等）
- 通知发送失败

### 11.2 错误处理策略
- 错误分类：可重试错误 vs 不可重试错误
- 指数退避重试策略（尤其对于网络和外部服务错误）
- 任务超时机制：防止任务无限期运行
- 检查点恢复：对于长任务，从最后检查点恢复
- 降级处理：如果知识点提取失败，可以尝试使用简化方法
- 人工干预队列：反复失败的任务移入特殊队列待人工处理
- 状态回滚：如果可能，回滚任务已做的更改以保持数据一致性

### 11.3 监控指标
- 任务提交速率和完成速率
- 各任务类型的平均执行时间
- 任务成功率和失败率
- 各种错误类型分布
- 爬虫成功率和平均响应时间
- 知识点提取速率和质量指标
- 题目生成速率和成功率
- 模型微调成功率和平均训练时间
- 数据同步速率和成功率
- 工作流调度延迟和漏任务率
- 资源使用率（CPU、内存、GPU、磁盘IO）
- 队列长度和平均等待时间
- API响应时间和错误率
- 任务延迟分布（实际执行时间 vs 计划执行时间）

### 11.4 日志策略
- 结构化日志便于日志系统解析
- 不同级别日志（DEBUG, INFO, WARN, ERROR）
- 关键操作日志（任务开始、完成、失败、重试）
- 性能日志（任务执行时间分布、资源使用趋势）
- 错误日志包含必要的上下文信息但避免记录敏感数据
- 审计日志记录任务创建、修改和删除的完整历史
- 爬虫日志记录访问的URL、状态码和响应时间
- 模型训练日志记录关键指标和epoch进度

## 12. 部署和扩展策略

### 12.1 容器化部署
- 使用Docker容器封装自学调度服务和工作流编排器
- 容器镜像分离：API服务容器和工作流执行容器可以不同
- 通过容器编排平台（Kubernetes）管理所有服务实例
- 使用Helm Charts或Kustomize简化部署和配置管理

### 12.2 扩展策略
- 水平扩展：
  - API服务：根据请求量增加实例数量
  - 工作流执行器：根据任务队列长度增加实例数量
  - 爬虫工作进程：根据爬虫任务并发需求增加数量
  - 模型训练工作进程：根据GPU可用性和训练需求增加数量
- 垂直扩展：根据监控调整单实例资源分配（模型训练可能需要更多内存和GPU）
- 动态扩展：基于队列深度和处理时间的自动伸缩策略
- 功能隔离爬虫：爬虫服务可以独立扩展而不影响其他功能
- 模型服务隔离：模型训练和模型服务可以分离部署

### 12.3 发布策略
- 蓝绿发布或滚动更新确保服务可用性
- 数据库向后兼容：新版本服务能够处理旧版本任务格式
- 工作流向后兼容：新版本定义的工作流能够在旧执行器上运行
- 配置版本控制：跟踪配置变更便于回滚
- 功能标志：逐步推出新功能而不影响现有用户
- 数据迁移策略：安全地升级数据库 schema 而不丢失数据

### 12.4 环境管理
- 开发、测试、预发布、生产环境隔离
- 环境变量管理：不同环境使用不同的配置
- 数据隔离：测试环境使用匿名化或虚拟数据
- 模型隔离：测试环境使用测试模型而不影响生产模型
- 第三方服务隔离：使用沙箱或测试账户访问外部服务

### 13.3 性能测试

- 大规模任务处理测试：测试处理大量任务的能力
- 爬虫性能测试：测试不同网络条件下的爬虫速度和成功率
- 知识点提取性能测试：测试不同内容长度和复杂度的处理时间

### 13.4 智能学习闭环测试
- 闭环功能测试：验证完整闭环能够正确执行
- 数据流完整性测试：验证数据在闭环各阶段之间正确传递
- 效果验证测试：验证闭环执行后能够提高模型性能或学习效果
- 参数敏感性测试：测试不同闭环参数对结果的影响
- 长期稳定性测试：验证闭环能够长期稳定运行而不退化

### 13.5 安全测试
- 输入验证测试：测试各种恶意输入的处理
- 认证和授权测试：确保未授权访问被正确拒绝
- 数据泄露测试：检查日志和错误信息中是否泄露敏感数据
- 爬虫安全测试：测试是否遵守目标网站规则和反爬虫策略
- 模型安全测试：测试是否存在提示词注入或模型滥用漏洞
- 数据安全测试：测试存储和传输过程中的数据保护
- 通知安全测试：测试是否存在通知滥用或垃圾信息发送风险
